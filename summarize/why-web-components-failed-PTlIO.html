Why Web Components Failed

I've been asked why I'm still using React when the browser now supports components, shouldn't everyone be using web components? Well, there's a reason why web components aren't gaining traction, and I'm not alone in thinking this. Notable figures like Rich Harris, the creator of Svelte, and Ryan Carniato, the creator of SolidJS, have expressed strong criticisms of web components.

Web Components Pose Significant Challenges

Ryan recently wrote an article titled "Web Components Are Not the Future," where he outlines the issues with web components. He acknowledges that while they promise interoperability and future-proofing, in reality, they introduce complexity and performance overheads. He even suggests that web components could "possibly pose the biggest risk to the future of the web" that he can see.

Elements Are Not Components

Web components are built on custom elements, but elements are not the same as components. Elements are a subset of components. Forcing every interface through the DOM with custom elements leads to problems. There are inconsistencies with attributes and properties, attributes accept only strings, while properties can handle any value. This mismatch complicates templating and requires additional prefixes or special handling in frameworks.

Standardization Limits Innovation

Standardizing high-level mechanisms like web components can stifle future innovation. Once something becomes a web standard, it's difficult to change. If the standard makes too many assumptions, it can prevent exploring alternative solutions. Ryan points out that "the introduction of higher-level primitives can have a positive effect," but warns that "it can have a negative effect if too many assumptions are made."

Performance Overhead and Complexity

Using web components introduces performance overhead due to the extra abstraction layer. They can negatively impact server-side rendering (SSR) because they rely on the DOM, which isn't present on the server. Frameworks have to implement workarounds, increasing the complexity and size of the codebase. This added complexity is a hidden tax that affects both developers and users.

The Community Consensus

The challenges aren't isolated incidents. Framework authors across the board have struggled with web components. Even React, which once considered web components, faced difficulties integrating them. The consensus among experienced developers is that web components may not be the ideal path forward.

* * *

While the idea of web components is appealing, creating reusable, interoperable components that work across different frameworks, the reality is that they introduce significant challenges. They complicate development, impose performance costs, and can hinder future innovation due to their rigid standardization. As Ryan puts it, **"Web components are compromised through and through." ** It's time we acknowledge these issues and focus on solutions that truly enhance the web development experience.
